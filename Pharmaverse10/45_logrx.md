## 1. Why execution logging matters

In SAS, every PROC writes to a `.log` file automatically. The log contains environment info, errors, warnings, notes — a complete record of what happened during execution. Regulators expect this log to be archived alongside any results.

In R, there's no native equivalent. A typical R script runs silently; you see output in the console or RStudio, but nothing persistent. For pharma's regulatory requirements — "prove what you ran and how" — this is a problem.

`{logrx}` fills the gap. It wraps an R script with comprehensive logging — capturing the environment, package versions, every line executed, results, errors, warnings. The output is a structured log file analogous to SAS's, suitable for archiving and audit.

Origin: developed by GSK, Atorus, J&J, and Pfizer collaboratively, released to CRAN ~2022. Active maintenance; current version 0.4.x as of mid-2026. Part of pharmaverse.

For pharma's R adoption story, **logrx is mandatory infrastructure**. Without it, R programs can't be audited the way SAS programs can.

## 2. Installation

```r
install.packages("logrx")
library(logrx)
```

## 3. The minimum-viable usage

The simplest invocation:

```r
axecute("my_script.R")
```

This:

1. Opens `my_script.R`
2. Captures environment info (R version, OS, user, working directory, loaded packages)
3. Executes the script line-by-line using `purrr::safely()` (so errors don't terminate the wrapping process)
4. Captures every output, warning, error
5. Writes a log file (`my_script.log` by default) with the captured information

After running, you have:

- The original R script unchanged
- A `.log` file beside it containing the execution record

For a typical ADaM build script, that's about a 10-minute setup, zero ongoing maintenance — and you've gained submission-grade traceability.

Every section is automatic. The information is sufficient to:

- Identify who ran the script and when
- Recreate the execution environment (R version, package versions)
- Detect any unauthorized package/function usage
- Trace any errors or warnings to their source
- Verify the script file itself wasn't modified (via hash)

For GxP audit purposes, this is essentially complete.

## 5. The `axecute()` function arguments

```r
axecute(
  file,                          # path to the R script
  log_name = NA,                 # log file name (default: same as script)
  log_path = NA,                 # log directory (default: same as script)
  include_rds = FALSE,           # also save log as .Rds for programmatic access
  quit_on_error = TRUE,          # exit with status 1 on error (CI-friendly)
  to_report = c("messages", "output", "result"),     # what to capture
  show_repo_url = FALSE,         # include CRAN/Posit Package Manager URLs
  extra_info = NA                # custom additional info to log
)
```

For typical use, defaults work. Common overrides:

- `log_path = "/var/log/r-studies"` for centralized logging
- `include_rds = TRUE` for tooling that consumes logs programmatically
- `quit_on_error = FALSE` for development (to continue after a failure)
- `extra_info = list(study = "STUDY001", env = "production")` for adding context

## 6. Running from the command line

For batch execution:

```bash
Rscript -e "logrx::axecute('ad_adsl.R')"
```

The `quit_on_error = TRUE` default (in CI mode) makes this play nicely with shell scripts:

```bash
#!/bin/bash
# Run all ADaM scripts; halt on first error

for script in programs/ad_*.R; do
  Rscript -e "logrx::axecute('$script')" || {
    echo "Error in $script. Halting."
    exit 1
  }
done
```

For Posit Connect scheduled jobs: just `Rscript -e "logrx::axecute(...)"` in the schedule.

## 7. Approved packages and functions

A pharma-specific feature: **approved-package lists**. Some organizations restrict which R packages and functions can be used in regulated programs. logrx supports this:

```r
# Define approved list (as a CSV or programmatically)
approved <- data.frame(
  package = c("dplyr", "dplyr", "admiral", "admiral", "xportr"),
  function_name = c("filter", "mutate", "derive_vars_merged", "derive_vars_dt", "xportr_write")
)

# Write to disk
write.csv(approved, "approved.csv", row.names = FALSE)

# Configure logrx to use it
options(log.rx.approved = "approved.csv")
```

Now when `axecute()` runs, any function used in the script that's not in the approved list gets flagged in the "Unapproved Packages and Functions" section of the log.

For sponsors implementing controlled-environment R adoption, this provides a soft enforcement mechanism: programmers see immediately if they've used an unapproved function. Hard enforcement comes from CI/CD checks rejecting builds with non-empty unapproved sections.

The `build_approved()` helper generates an approved list from a directory of validated scripts:

```r
build_approved(directory = "validated_scripts/")
```

## 8. Generating `renv` lockfiles from logs

A useful logrx feature: the log contains exact package versions used during execution. You can convert this into an `renv` lockfile for reproducing the environment:

```r
# From the rendered log
read_log_file("ad_adsl.log") |>
  generate_lockfile(path = "renv.lock")
```

This produces an `renv.lock` file pinning the exact package versions. Anyone else running `renv::restore()` in their project gets the same versions.

For reproducibility, this closes a critical gap: the log proves what was run; the lockfile lets others recreate the environment. Together they form a complete reproducibility record.

## 9. The log_*() function family

For advanced use, the underlying log functions are exported:

| Function | Purpose |
|---|---|
| `log_init()` | Initialize the log object |
| `log_config()` | Add environment info (auto-called by axecute) |
| `log_write()` | Write the log to disk |
| `log_remove()` | Clean up the log object |
| `read_log_file()` | Parse an existing log file into a structured object |
| `write_log_header()` | Add header info to a log |
| `write_used_functions()` | Add the "used functions" section |
| `write_unapproved_functions()` | Add the "unapproved functions" section |

For typical use, `axecute()` orchestrates these. The log_*() family is for custom workflows where `axecute()` doesn't fit (e.g., logging interactive sessions, partial-script logging).

## 10. Logging Rmd and Quarto files

`{logrx}` works on `.Rmd` and `.qmd` files too:

```r
axecute("study_report.Rmd")
```

Behavior: renders the Rmd, captures the rendering (knit, output), and produces a log alongside the rendered output (typically `.html` or `.pdf`).

For CSR-style Rmd reports that need to be reproducible, wrapping the render with axecute provides full traceability — the log shows package versions and execution environment used to produce the report.

## 11. Integration with shiny / teal

For Shiny apps, logging is more nuanced because the "execution" is event-driven, not linear. Two patterns:

**Pattern A — log the deployment, not the runtime**: when the Shiny app is deployed to Posit Connect, the deployment uses `axecute()` to capture the environment. Per-user app interactions are logged separately by Posit Connect itself.

**Pattern B — `{teal.logger}`**: teal has a built-in logger (`{teal.logger}`) that captures user inputs across modules. Different from logrx (event-based vs script-based), but complementary.

For most teal apps, Pattern A is sufficient: the package environment is captured at deployment via logrx, and user interactions are logged by Posit Connect's audit trail. logrx isn't the right tool for inside-app event logging.

## 12. Comparison to alternatives

Other R logging packages exist; how does logrx compare?

| Package | Focus | logrx differentiator |
|---|---|---|
| `{log4r}` | Generic logging (info, warn, error levels) | Not pharma-specific; needs manual configuration |
| `{logger}` | Modern generic logging | Not pharma-specific |
| `{whirl}` | Pharma-focused (Novo Nordisk) | More feature-rich for complex workflows; less mature CRAN presence |
| `{logrx}` | Pharma-focused (GSK, Atorus, J&J, Pfizer) | Mature, CRAN, axecute() one-liner UX |

For the "I just want to log a script execution for compliance" use case, **logrx is the simplest and most pharma-aligned**. The `axecute()` one-liner UX matters.

The pharmaverse examples page maintains a side-by-side comparison of logrx, whirl, and other options. Worth checking when evaluating for your team.

## 13. Putting it together: a logged ADaM build

A complete logged ADaM build script:

```r
# programs/ad_adsl.R

library(admiral)
library(dplyr)
library(metacore)
library(metatools)
library(xportr)

# Load spec and source data
mc <- metacore::spec_to_metacore("specs/adam_spec.xlsx")
dm <- haven::read_xpt("sdtm/dm.xpt")
ex <- haven::read_xpt("sdtm/ex.xpt")

# Build ADSL
adsl <- dm |>
  derive_vars_merged(
    dataset_add = ex |> filter(EXSEQ == 1),
    new_vars = exprs(TRTSDT = EXSTDT),
    by_vars = exprs(STUDYID, USUBJID)
  ) |>
  mutate(SAFFL = if_else(!is.na(TRTSDT), "Y", "N")) |>
  drop_unspec_vars(metacore = mc, dataset = "ADSL") |>
  check_variables(metacore = mc, dataset = "ADSL") |>
  apply_variable_labels(metacore = mc, dataset = "ADSL")

# Write XPT
adsl |>
  xportr_metadata(mc, "ADSL", verbose = "warn") |>
  xportr_type() |>
  xportr_length() |>
  xportr_label() |>
  xportr_order() |>
  xportr_format() |>
  xportr_df_label(mc) |>
  xportr_write("submission/adsl.xpt")

# Save R version as .rds for traceability
saveRDS(adsl, "outputs/adsl.rds")
```

Wrap with logrx:

```bash
Rscript -e "logrx::axecute('programs/ad_adsl.R')"
```

Results:

- `submission/adsl.xpt`: the FDA-ready XPT
- `outputs/adsl.rds`: R-readable archive
- `programs/ad_adsl.log`: complete execution log

The `.log` file shows:

- Who ran it (programmer1)
- When (2026-05-22 09:15:23 UTC)
- R version, OS, all loaded package versions
- Every admiral/xportr/dplyr function used
- Any warnings or errors
- Final result

For audit: archive the script + log + lockfile together. Anyone reviewing can verify what was executed and recreate the environment.

## 14. The pharma directory pattern

A standard pharma R-programs directory structure:

```
study_X/
├── data/
│   ├── sdtm/
│   └── adam/
├── programs/
│   ├── ad_adsl.R
│   ├── ad_adae.R
│   ├── ad_adlb.R
│   └── ...
├── logs/
│   ├── ad_adsl.log
│   ├── ad_adae.log
│   └── ...
├── outputs/
│   └── (RTF files from TLG scripts)
├── specs/
│   └── adam_spec.xlsx
├── renv.lock
├── run_all.sh
└── README.md
```

The `run_all.sh` typically iterates programs/ with logrx:

```bash
#!/bin/bash
set -e

for script in programs/*.R; do
  echo "Running: $script"
  Rscript -e "logrx::axecute('$script', log_path = 'logs/')"
done

echo "All programs executed successfully."
```

This pattern is standard at Atorus-aligned sponsors and increasingly broadly. The output is reproducible: re-running the script should produce identical XPTs and identical logs (modulo timestamps).

## 15. CI/CD integration

For GitLab CI / GitHub Actions running R-based pharma pipelines:

```yaml
# .gitlab-ci.yml or similar
adam_build:
  script:
    - R -e "renv::restore()"
    - for f in programs/*.R; do R -e "logrx::axecute('$f')"; done
    - ls -la logs/
    - test ! -s logs/*errors  # fail if any errors
  artifacts:
    paths:
      - logs/
      - submission/
      - outputs/
```

The CI/CD pipeline:

1. Restores the R environment via renv
2. Runs each ADaM build script with logrx
3. Archives the logs and outputs as build artifacts
4. Fails the build if any log shows errors

For sponsors with mature R DevOps practices, this is standard. Every pipeline run produces a complete, archived audit trail.

## 16. Auditing logs programmatically

The `read_log_file()` function parses a log into a structured object:

```r
log_data <- read_log_file("ad_adsl.log")
str(log_data)

# Example: extract all functions used
log_data$used_functions

# Example: check for any errors
length(log_data$errors) == 0
```

For QC programs that audit production runs:

```r
# Audit script: check all logs for issues
for (log_file in list.files("logs/", pattern = "\\.log$", full.names = TRUE)) {
  log_data <- read_log_file(log_file)
  if (length(log_data$errors) > 0) {
    cat("ERRORS in", log_file, ":\n", paste(log_data$errors, collapse = "\n"), "\n")
  }
  if (length(log_data$unapproved_packages) > 0) {
    cat("UNAPPROVED packages in", log_file, ":", log_data$unapproved_packages, "\n")
  }
}
```

For sponsor QA roles: build dashboards that show "across all studies, are any production logs flagging issues?" Aggregate auditing becomes possible.

## 17. Common pitfalls

### "My script doesn't produce a log"
Cause: `axecute()` wasn't called, or it errored before writing the log.
Fix: check the working directory; the log should be alongside the script unless `log_path` was specified.

### "Log shows hundreds of unapproved functions"
Cause: approved-packages list isn't configured or is incomplete.
Fix: build a comprehensive approved list via `build_approved()` from a known-good directory, or accept that unapproved-tracking isn't your priority.

### "Log is huge"
Cause: script produces extensive output captured by logrx.
Fix: use `to_report = c("messages")` to skip output capture, or restructure the script to produce less verbose output.

### "Log timestamp doesn't match production"
Cause: time zone mismatch between R session and audit expectations.
Fix: set `Sys.setenv(TZ = "UTC")` at the start of scripts, or document the conversion convention.

### "Different runs produce different logs"
Cause: non-deterministic operations (random sampling, system time embedded in output).
Fix: set `set.seed()` for any random operations; exclude timestamps from comparison.

## Glossary

- **`{logrx}`** — Cross-pharma R script execution logging package
- **`axecute(file)`** — One-liner that wraps script execution with full logging
- **`log_init()` / `log_config()` / `log_write()` / `log_remove()`** — Lower-level log functions
- **`read_log_file()`** — Parse an existing log into structured R object
- **`generate_lockfile()`** — Produce renv.lock from logrx log
- **`build_approved()`** — Generate approved-packages list from a directory of scripts
- **Approved packages list** — Controlled-environment whitelist of allowed packages/functions
- **`renv`** — R environment management; pairs with logrx for full reproducibility
- **`extra_info`** — Argument for adding custom context to the log (added 0.4.0)
- **`quit_on_error`** — Exit with status 1 on error; CI-friendly
- **`to_report`** — Controls what gets captured: messages, output, result
- **`{teal.logger}`** — Complementary teal-specific event logger; not a replacement for logrx
- **`{whirl}`** — Alternative pharma logging package by Novo Nordisk
- **SAS .log file** — The traditional analog logrx is recreating for R
- **GxP audit** — The regulatory inspection logrx enables you to pass
